# 三国杀游戏自动任务脚本设计文档

## 1. 文档目的

本文档用于设计一个“三国杀游戏自动任务脚本”的工程方案，目标是帮助开发者在合法、授权、低风险的前提下完成重复性任务辅助、流程验证和个人效率工具建设。

本设计不包含也不支持以下能力：

- 绕过游戏登录、验证码、风控、反作弊或安全校验
- 修改游戏客户端、内存、封包、服务端数据或对局结果
- 在竞技、排位、匹配、真人对局中进行无人值守自动化
- 批量养号、刷资源、刷活动、刷奖励或其他违反平台规则的行为

推荐使用场景：

- 本地开发环境或授权测试环境中的 UI 自动化验证
- 个人账号在明确允许的范围内执行非竞技、低影响的重复操作
- 游戏界面识别、任务状态记录、操作流程回放等学习型工具

## 2. 项目目标

### 2.1 核心目标

- 自动识别当前页面或游戏界面状态
- 根据预设任务清单执行有限步骤
- 记录任务执行日志、失败原因和截图证据
- 提供可配置的任务开关、执行间隔和安全停止机制
- 支持人工接管，避免脚本在异常状态下持续操作

### 2.2 非目标

- 不追求“全天候无人值守”
- 不实现任何规避检测能力
- 不处理真实对局中的出牌策略自动决策
- 不通过非公开接口调用游戏服务

## 4. 模块设计

### 4.1 配置层

配置层负责声明脚本允许执行的任务、运行限制和识别素材。

建议配置项：

```json
{
  "profile": "personal-local",
  "maxRunMinutes": 20,
  "taskIntervalMs": 1200,
  "maxRetriesPerTask": 2,
  "requireManualConfirm": true,
  "tasks": [
    {
      "id": "daily-check",
      "enabled": true,
      "mode": "non-competitive",
      "steps": ["open-task-panel", "claim-available-reward"]
    }
  ]
}
```

设计要求：

- 所有任务默认关闭，由用户显式启用
- 每个任务必须声明适用场景
- 高风险操作必须要求人工确认
- 配置文件不保存账号密码、令牌或敏感凭据

### 4.2 任务调度器

任务调度器负责按顺序执行任务，并处理失败、重试和退出。

职责：

- 加载任务配置
- 校验任务是否允许执行
- 按队列执行任务步骤
- 控制重试次数
- 在超时、异常、人工暂停时停止

建议状态：

- `idle`：空闲
- `running`：执行中
- `paused`：用户暂停
- `failed`：失败退出
- `completed`：任务完成

### 4.3 状态识别模块

状态识别模块用于判断当前界面是否符合下一步操作条件。

可选识别方式：

- DOM 识别：适用于网页版本或可访问 DOM 的容器
- 图像模板匹配：适用于固定按钮、图标、弹窗
- OCR 文本识别：适用于任务标题、奖励状态、提示语
- 坐标区域识别：仅作为兜底方案，不作为主要依据

识别原则：

- 优先使用稳定、可解释的信号
- 每次操作前确认页面状态
- 操作后重新识别状态，而不是盲目连续点击
- 识别置信度低于阈值时停止并请求人工确认

### 4.4 操作执行模块

操作执行模块负责执行鼠标、键盘或页面级动作。

支持动作：

- 点击指定按钮
- 输入文本
- 等待页面变化
- 滚动任务列表
- 截图留证

限制要求：

- 不支持后台隐藏执行关键动作
- 不支持高频点击
- 不支持跨越安全提示、验证码或风控弹窗
- 不支持绕开用户确认的奖励领取或消耗类操作

### 4.5 安全控制器

安全控制器是脚本的强制约束层。

必须能力：

- 一键停止
- 最大运行时长
- 最大点击次数
- 异常弹窗停止
- 低置信度停止
- 任务完成后自动退出

建议阈值：

```json
{
  "maxRunMinutes": 20,
  "maxClicksPerMinute": 20,
  "minRecognitionConfidence": 0.82,
  "stopOnUnknownDialog": true,
  "stopOnCompetitiveMode": true
}
```

### 4.6 日志与审计模块

日志模块用于回溯脚本行为，便于调试和风险排查。

日志内容：

- 启动时间、结束时间
- 当前任务 ID
- 每一步识别结果
- 每一次动作
- 失败原因
- 关键截图路径

日志示例：

```text
2026-06-06 11:50:21 task=daily-check step=open-task-panel status=ok
2026-06-06 11:50:23 task=daily-check step=claim-available-reward status=manual-confirm-required
2026-06-06 11:50:28 task=daily-check status=completed
```

## 5. 任务流程设计

### 5.1 启动流程

```text
读取配置
  -> 校验运行环境
  -> 显示任务列表
  -> 用户确认
  -> 启动安全控制器
  -> 执行任务队列
```

### 5.2 单任务执行流程

```text
进入任务
  -> 识别当前界面
  -> 判断是否满足步骤条件
  -> 执行动作
  -> 等待界面稳定
  -> 重新识别
  -> 记录日志
  -> 完成或停止
```

### 5.3 异常处理流程

```text
发现未知状态
  -> 截图
  -> 写入日志
  -> 暂停任务
  -> 提示用户接管
```

## 6. 数据结构设计

### 6.1 Task

```ts
type Task = {
  id: string;
  name: string;
  enabled: boolean;
  mode: "local-test" | "non-competitive";
  requiresManualConfirm: boolean;
  steps: TaskStep[];
};
```

### 6.2 TaskStep

```ts
type TaskStep = {
  id: string;
  description: string;
  detect: DetectionRule;
  action: ActionRule;
  timeoutMs: number;
  retryLimit: number;
};
```

### 6.3 DetectionRule

```ts
type DetectionRule = {
  type: "dom" | "image" | "ocr";
  target: string;
  confidence?: number;
  region?: {
    x: number;
    y: number;
    width: number;
    height: number;
  };
};
```

### 6.4 ActionRule

```ts
type ActionRule = {
  type: "click" | "type" | "wait" | "scroll" | "screenshot";
  target?: string;
  value?: string;
  delayMs?: number;
};
```

## 7. 技术选型建议

### 7.1 桌面自动化方案

适用场景：Windows 桌面客户端或模拟器窗口。

候选技术：

- Python + PyAutoGUI：简单易用，适合原型
- OpenCV：图像模板匹配
- PaddleOCR / Tesseract：文字识别

优点：

- 上手快
- 对普通窗口兼容性较好

缺点：

- 坐标和分辨率敏感
- 需要更多安全停止逻辑

### 7.2 Web 自动化方案

适用场景：网页版本或可被浏览器控制的游戏界面。

候选技术：

- Playwright
- Puppeteer

优点：

- 状态识别更稳定
- 日志和截图能力成熟

缺点：

- 仅适用于可自动化的网页环境
- 需要遵守网站规则和授权边界

### 7.3 推荐路线

第一阶段建议使用“可视化识别 + 人工确认”的桌面自动化原型：

- 不接触游戏内部数据
- 不修改客户端
- 不执行竞技场景动作
- 所有关键步骤保留人工确认

## 8. 风险与合规

### 8.1 规则风险

游戏厂商通常禁止自动化脚本、外挂、代练、刷资源等行为。即使脚本只做重复操作，也可能违反用户协议。

控制措施：

- 在使用前阅读并遵守平台规则
- 不在竞技、匹配、排位或真人对局中使用
- 不做奖励刷取、资源刷取或账号批量操作
- 不绕过任何安全机制

### 8.2 账号风险

自动化行为可能导致账号限制、封禁或数据异常。

控制措施：

- 使用测试账号或授权环境
- 设置运行时长和频率上限
- 保留日志和截图
- 发生异常立即停止

### 8.3 安全风险

脚本可能误点消耗道具、确认弹窗或隐私信息。

控制措施：

- 消耗类操作强制人工确认
- 未知弹窗立即停止
- 禁止存储账号密码
- 禁止自动提交敏感信息

## 9. 开发计划

### 阶段一：原型验证

- 建立配置文件
- 实现任务调度器
- 实现截图和日志
- 实现人工确认机制
- 完成一个本地演示任务

### 阶段二：状态识别

- 增加图像模板匹配
- 增加 OCR 识别
- 增加置信度阈值
- 增加未知状态暂停

### 阶段三：任务编排

- 支持多任务队列
- 支持失败重试
- 支持任务启停配置
- 支持执行报告

### 阶段四：安全加固

- 增加运行时长限制
- 增加点击频率限制
- 增加敏感操作确认
- 增加审计日志导出

## 10. 验收标准

- 脚本只能执行配置中显式启用的任务
- 未知界面、低置信度识别和异常弹窗会停止执行
- 所有关键动作有日志记录
- 消耗类、奖励类或不可逆操作需要人工确认
- 不包含绕过风控、反作弊或客户端保护的实现
- 可以在测试环境完成一次完整任务流程演示

## 11. 后续扩展

- 图形化任务配置面板
- 任务执行报告页面
- 识别素材管理
- 多分辨率适配
- 本地测试回放工具

## 12. 结论

该脚本应被定位为“授权场景下的流程辅助与测试工具”，而不是在线游戏外挂或无人值守刷取工具。设计重点应放在可控、可停、可审计和低风险上，所有可能影响公平性、账号安全或平台规则的能力都应从实现范围中排除。
